# live offset、播放器缓冲与磁盘缓存关系图

这张图用来说明当前项目里 3 层机制的关系：

- HLS 直播窗口
- ExoPlayer 自身的播放缓冲
- `HlsSegmentPrefetcher` 的磁盘预取缓存

## 关系总览

```text
直播源
  -> m3u8 持续刷新
  -> 当前窗口: [361, 362, 363, 364, 365]
                     ^
                     最新分片 / live edge

ExoPlayer
  -> 尽量维持在 target live offset 附近播放
  -> 维护自己的内存 buffer
  -> 持续向后请求新的 ts 分片

HlsSegmentPrefetcher
  -> 当播放器实际读取到 X 时
  -> 尝试预取 X+1 / X+2 / X+3
  -> 把预取结果写入磁盘缓存
```

## 正常命中的时序

当播放器和 live edge 之间还有足够提前量时，磁盘缓存可以先写好，再被播放器读取：

```text
t0
播放器正在播放 365

t0
预取器开始下载 366 / 367 / 368

t1
366 预取完成，写入磁盘缓存

t2
播放器推进到 366
-> 命中磁盘缓存
-> 不需要重新回源下载
```

这种情况下：

- ExoPlayer 自己的 buffer 负责“当前播放不断流”
- 磁盘缓存负责“未来分片提前准备好”

## 当前日志里常见的情况

当前项目分析日志后，更常出现的是下面这种：

```text
t0
播放器正在播放 365

t0
预取器开始下载 366

t0 + 很短时间
播放器马上就需要读取 366

t0 + 很短时间
366 还没预取完成
-> 取消预取
-> 播放器直接回源读取 366
```

这时日志里通常会看到：

- `Prefetch start`
- `Playback needs current segment now`
- `Playback bypass cache for in-flight segment`
- `Prefetch cancelled`

## 为什么会这样

这不是因为播放器“已经缓冲到 18 秒就不再读取新片”。

恰恰相反，直播播放会持续向前推进：

- 播放器即使已经在某个 live offset 附近开始播放
- 仍然会继续读取后续新进入窗口的 ts 分片
- 预取缓存一旦写好，也确实可能马上就被播放器读到

真正的问题是：

```text
不是缓存写好后读得太快
而是很多时候还没写完，播放器就已经要读了
```

## 三者的职责边界

### 1. HLS 直播窗口

- 决定当前能看到哪些 ts 分片
- 最新一片就是当前 live edge
- 窗口越短，播放器和未来分片之间的操作空间越小

### 2. ExoPlayer 自身缓冲

- 主要是内存 buffer
- 目标是让“当前播放不要卡顿”
- 会持续向后请求分片，不会因为已经开始播放就停止拉流

### 3. `HlsSegmentPrefetcher` 磁盘缓存

- 目标是提前下载未来几个分片
- 如果下载先完成，后续播放可直接命中磁盘缓存
- 如果播放器先到，预取任务会被抢占取消

## 优先级关系

当前实现里的优先级是：

```text
当前播放可用性 > 预取任务完成
```

也就是说：

- 只要播放器马上需要当前分片
- 预取任务就必须让路
- 本次播放直接回源，避免等待未完成的缓存

## 对当前项目的实际含义

当前日志说明：

- 磁盘缓存机制本身是有效的
- 但在播放器过于贴近 live edge 时，很多 `X+1` 分片没有足够时间完成预取
- 因此命中率会明显下降

所以当命中率低时，首先要判断的不是“磁盘缓存有没有用”，而是：

- 当前播放位置是否离 live edge 太近
- 新分片进入窗口后，预取是否有足够提前量

## 一句话总结

```text
ExoPlayer buffer 解决“现在别卡”
磁盘预取缓存解决“下一片提前准备好”
当下一片已经快变成“现在这一片”时，播放器会直接抢占预取任务
```
